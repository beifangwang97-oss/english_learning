# CODEX 项目日志（整理版）
更新时间：2026-04-07

## 1. 会话目标与范围
- 持续完善管理员端、教师端、学生端的账号与学习流程联动。
- 当前重点：
1. 教师端权限管理（仅本门店学生）
2. 管理员端账号管理与教师端数据一致性
3. 登录状态严格依赖数据库（启用状态 + 截止日期）
4. 文案与编码规范稳定执行，避免乱码与语言回退

## 2. 已确认的业务规则
- 教师端账号管理能力与管理员端“字段内容”保持一致，但教师仅可操作本门店学生。
- `storeName` 不允许被教师修改。
- 教师端支持：
1. 单行编辑、单行保存
2. 单个删除与批量删除（仅本门店学生）
3. “添加学生账号”（弹窗录入，不使用 CSV/Excel）
- 手机号可作为唯一标识，若已存在必须报错，阻止写入。
- 字段强校验：新增/编辑必须完整覆盖关键字段，不允许出现关键字段缺失（如截止日期为空）。
- 管理员端与教师端使用同一份数据；修改后另一端手动刷新可见。

## 3. 登录与账户状态规则（强制）
- 登录校验严格依赖数据库实时状态。
- 若 `active = false`，登录必须被拒绝，并返回“账号已停用”。
- 若 `expireDate` 到期（或为空，按系统规则视为不可登录），登录必须被拒绝，并返回“账号已到期”。
- 禁止仅依赖前端缓存状态判断登录可用性。

## 4. 已完成的关键改动（摘要）
- 后端用户服务已加强：
1. 登录时校验启用状态与截止日期
2. 新增/编辑时加强关键字段完整性约束
- 教师端权限管理页已改为真实接口驱动（非 mock）。
- 管理员端账号管理页已修复为中文文案，避免英文回退。
- 已执行前端构建验证：`front -> npm run build` 通过。

## 5. 运行与联调要求
- 每次进入新会话，先执行：
1. 阅读本日志
2. 验证 `run_all` 启动后的端口监听与关键接口可用性
- 若关键服务未启动或端口异常，先修复运行环境，再继续功能开发。

## 6. 关键接口（账号与权限相关）
- 管理员端用户管理：查询、创建、更新、删除（全局权限）。
- 教师端用户管理：查询、创建、更新、删除（仅本门店学生）。
- 登录接口：返回结果必须体现数据库实时状态校验结论。

## 7. 提交流程要求
- 修改账号相关功能后必须做最小闭环验证：
1. 教师端新增/编辑/删除（含批量）学生账号
2. 管理员端刷新后数据同步可见
3. 启用状态关闭或截止日期到期后，登录被拒绝
- 未完成验证不得标记为完成。

## 8. 文档与编码规范（强制，唯一版本）
- 仓库文本文件统一使用 UTF-8（无 BOM）。
- Windows 终端下编辑中文内容时，禁止使用依赖本地代码页的覆盖方式，避免乱码。
- 平台面向中文用户，前端可见业务文案默认使用中文。
- 除接口字段、协议字段、第三方专有名词外，不得将可见业务文案改为英文。
- 提交前必须检查：按钮、表头、提示、弹窗、校验信息均为中文且无乱码。
- 若发现乱码或英文文案回退：先修复文案与编码问题，再继续开发。
- 每次新会话开始时，默认先读取并执行本节规范。
